Automatic Fee Application
Figma Design Link
What?
Provision of adding multiple pricing details with their applicability date. Additionally, for each pricing detail, have the option for making it as automatic by specifying cases or either inclusions or exclusions.
Why?
We don't have the option to tag different fee details with their applicability date and pay types under one Fee Code. Along with that, we also don't have the option to add markup fee automatically when we sell a part.
How?
Ability to add multiple pricing details in the Franchise Pricing section when we add or edit a fee code. We also have a toggle for automatic fee application. When turned on, we can either choose to define inclusions or exclusion cases by specifying the customers, source codes and parts.

Parts Integration Dashboard (Internal)
Figma Design Link
What?
OEM send parts master updates via different file formats. Tekion's internal teams are responsible for timely and quality updates to parts master and local inventory. There are different stages through which the integration file goes through. This design enables the internal stakeholders to track where is the file stuck in the whole integration journey.
Why?
On account of delays in part master updates, the dealership blames Tekion as a platform for the delays. In reality, the reason can be many - OEM's delay in sending files or corrupt file sent or the file is stuck in any of the stages in the integration journey. Today, it's difficult to track what is causing the delay. This design helps in troubleshooting.
How?
By introducing a dashboard for internal stakeholders and informing them timely so that action can be taken at the right time, to prevent integration delays

Automatic Source Code Movements
Figma Design Link
What?
An ability to have the intelligence in the system to move a part from once source code to another when they meet a specific criteria either realtime or at a specific trigger ( say master update)
Why?
Source Code of a part is a critical functionality to drive the parts operations at a DP. Source codes drive a lot of decisioning particularly around default pricing as well as the storage and ordering of parts. Source Codes are also used for various other purposes to group a particular set of parts with a certain characteristics.
How?
Under OEM settings, allowing the users with an option for automatic source code movements option with
  1. Ability to define one rule as a combination of multiple fields/combinational logic
  2. Ability to configure more than one criteria/rules for auto-source code movements
  3. Ability to run this auto movements on demand, factory master update and addition of new part
  4. Ability to prioritise amongst the criteria in the rule engine by simple drag and drop ordering
